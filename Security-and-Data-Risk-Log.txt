SECTION 1 — GENERAL OPERATING PRINCIPLES
=======================================================

Rocky operates under a security-first, privacy-first, read-only-by-default posture.

Core principles applied to all integrations:
- Least privilege: request only the minimum permissions needed for the specific task
- Explicit approval: write/modify actions require explicit authorization before execution
- Credential hygiene: no secrets in code, commits, or chat logs; stored in encrypted config only
- Data minimization: summaries and aggregates preferred over raw records
- Audit trail: all connections, permissions, and decisions logged in this document

Hard limits are documented separately in: Operation Hard Limits.txt

SECTION 4 — ZOOM API (Contact Center + Reports)
=======================================================

Date: 2026-03-26
App type: Server-to-Server OAuth (no user login required)

Permissions granted (all read-only):
- Reports: sign-in/out activity, operation logs, user reports, daily usage
- Contact Center: engagements, queues, agents, recordings, skills, dispositions,
  flows, teams, performance datasets, roles

Risk considered:
- API has read access to sensitive call data including client phone numbers,
  call recordings, and agent activity
- Call recordings contain protected client communications

Mitigations applied:
- All scopes are read-only — no ability to modify Zoom settings or data
- Recording download URLs require valid API bearer token — not publicly accessible
  without Rocky's credentials
- Recording proxy built (Vercel) to allow Matt to download recordings without
  exposing API credentials to the browser
- Proxy password-protected; credentials stored server-side in Vercel encrypted env vars

Decision on recording access delivery (2026-03-27):
- Option considered: Post recording links in Discord
- Risk identified: Discord links could be copied and shared outside the auth wall
- Option considered: Zoom Team Chat delivery
- Research finding: Zoom Team Chat does NOT support per-channel retention policies
  (account-level only)
- Decision: Post recording download links in a private Zoom Team Chat channel
  restricted to Matt Dugan and Ally Stratos only
- Rationale: Both users already have admin access to recordings via Zoom web portal;
  the delivery channel restricts visibility without expanding access
- Status: Pending — Zoom Team Chat scopes not yet added to Rocky's app

Residual risk: MODERATE (pending) — recording links currently delivered via Discord.
Will be reduced to LOW once Zoom Team Chat delivery is implemented.

Zoom Phone scopes added: 2026-03-28
- phone:read:list_call_logs:admin — read-only access to phone call logs
- Purpose: look up calls by number/extension when Contact Center search returns no results

App name: Rocky AI Assistant

Mitigations applied — Email/Calendar:

- Security group created: rocky-ai-access@slsct.org (contains only the two accounts)

- Client secret expiry: 2028-03-28 — calendar reminder set in mdugan@slsct.org

  - Hidden from tenant address book
  - Search indexing disabled (content not indexed in tenant search)
  - External sharing: organization members only
- Credential drop process established: Matt uploads credential files to SharePoint;
  Rocky reads, saves to TOOLS.md, and deletes the file — credentials never in Discord

Residual risk: LOW — multiple layers of restriction applied at both app and site level.

SECTION 9 — ALLY COLLABORATION, WORDPRESS / ECHO KB, AND TASK-SPECIFIC APPROVALS
=======================================================

Date: 2026-04-10
Decision: Allow Rocky to collaborate directly with Ally in a dedicated Discord channel,
use approved access to Ally systems for bounded tasks, and perform document posting work
into the SLS WordPress wiki via Echo Knowledge Base.

Risk considered:
- Group-channel identity ambiguity could cause Rocky to treat a message as Ally when it is not
- Mailbox access to Ally's inbox introduces sensitive read exposure beyond Matt's mailbox
- WordPress / Echo KB write actions could alter live staff, board, or attorney knowledge-base content
- Uploading non-PDF files to the wiki causes functional/usability issues in the KB experience
- Echo KB permission controls could be changed accidentally if workflow boundaries are not explicit

Mitigations applied:
- Matt instructed Rocky to treat the dedicated Ally Discord channel as Ally by default, while still verifying sensitive or write approvals when needed
- Matt explicitly approved a task-specific exception allowing Rocky to access Ally's mailbox for the `Document for WIKI` task and upload the attached document to the Rocky Test knowledge base
- WordPress work was tested first in `Rocky Test` (`epkb_post_type_4`) rather than directly in the production-facing knowledge bases
- Rocky confirmed the working Echo KB process:
  1. retrieve the source document
  2. convert to PDF first if the source file is not already a PDF
  3. upload the PDF to the WordPress Media Library
  4. create or update a custom link article in the target KB
  5. assign the correct category
- New hard-limit rule recorded separately: never delete WordPress articles or pages in this Echo KB workflow
- Additional workflow rule recorded separately: do not change Echo KB user/group/page access permissions during routine document-posting work unless explicitly approved

Implementation details observed:
- Microsoft Graph application access was used to search Ally's mailbox and retrieve the task email attachment
- WordPress interactions were performed through a local scripted authenticated web session as the `rocky` user
- Local Python processing was used for document extraction and conversion to PDF before final wiki upload
- The first live workflow succeeded end-to-end in Rocky Test after Ally updated WordPress permissions for the `rocky` user

Residual risk: LOW-MODERATE — mailbox and wiki access remain sensitive, but were bounded by
explicit task approval, test-knowledge-base use, PDF-only workflow normalization, and
clear no-delete/no-permission-change constraints.
